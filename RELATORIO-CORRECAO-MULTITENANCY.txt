📋 RESUMO EXECUTIVO
═══════════════════════════════════════════════════════════════════════════════

❌ PROBLEMA ENCONTRADO:
   Todas as queries de dados de clientes/vendas tinham fallback perigoso:

   req.tenantId || 1

   Se req.tenantId fosse undefined, a query operaria SILENCIOSAMENTE no tenant 1,
   vazando dados de todo cliente/admin para outro.

   Impacto:
   • Um vendedor poderia ver TODOS os clientes (dados de outros clientes)
   • Um vendedor poderia registrar vendas no tenant 1 (dados de admin)
   • Relatórios mostrariam dados misturados
   • Zero avisos ou logs de erro


✅ SOLUÇÃO IMPLEMENTADA:

   1️⃣  Removidas 157 ocorrências de req.tenantId || 1
   2️⃣  Adicionada validação explícita (if (***REMOVED***req.tenantId) return 401)
   3️⃣  Middleware global que bloqueia qualquer rota protegida sem tenantId
   4️⃣  Teste de segurança + Documentação

═══════════════════════════════════════════════════════════════════════════════
🛡️ MECANISMO DE PROTEÇÃO
═══════════════════════════════════════════════════════════════════════════════

Em server.js, adicionado NOVO middleware GLOBAL:

    app.use('/api', injetarTenant);        // Injeta req.tenantId da sessão
    app.use('/api', garantirTenantId);     // 🔒 BLOQUEIA se tenantId faltar
    app.use('/api', validarTenantAtivo);   // Verifica se tenant está bloqueado

Ordem:
  1. injetarTenant: req.tenantId = session.tenant_id (ou 1 se admin)
  2. garantirTenantId: SE (***REMOVED***req.tenantId) return 401 ← 🔒 BARREIRA
  3. validarTenantAtivo: verifica se tenant está bloqueado
  4. Rota protegida: usa req.tenantId com garantia de validação
